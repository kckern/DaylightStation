# NF-1 + NF-5 Bugfix Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Fix two silent correctness failures found by the post-audit independent review — shuffle mode displays the wrong current track (NF-1), and the device monitoring broadcast never fires because `usePlaybackBroadcast` holds a dead ref (NF-5).

**Architecture:** NF-1 is a data bug in `useMediaQueue` — `shuffleOrder` is stripped from WebSocket-synced state, and `currentItem` ignores it entirely. NF-5 is a structural wiring bug — the context creates a `playerRef` that is never attached to the Player DOM, while `NowPlaying` creates its own local ref that IS attached but is not shared upward to the broadcast hook.

**Tech Stack:** React 18, Vitest + React Testing Library (hook tests), no new dependencies.

---

## The Two Bugs

### NF-1: shuffleOrder stripped from state → wrong currentItem in shuffle mode

`useMediaQueue.js` holds queue state as a plain JS object. When the WebSocket syncs a `media:queue` broadcast, `handleQueueBroadcast` reconstructs state field-by-field but **omits `shuffleOrder`**. Meanwhile, `currentItem` is computed as `items[position]` — ignoring `shuffleOrder` entirely. The backend's `MediaQueue.currentItem` getter computes `items[shuffleOrder[position]]` when shuffle is on. So when shuffle is enabled, frontend and backend disagree on which item is "current."

**Evidence:**
- `useMediaQueue.js:51-57` — `handleQueueBroadcast` sets items/position/shuffle/repeat/volume. No `shuffleOrder`.
- `useMediaQueue.js:164-167` — `currentItem = useMemo(() => items[position], ...)`. No shuffleOrder check.
- `MediaQueue.mjs:74-75` — Backend: `const idx = this.shuffle && this.shuffleOrder.length > 0 ? this.shuffleOrder[this.position] : this.position;`

### NF-5: context playerRef is never attached to the Player

`MediaAppContext` creates `playerRef = useRef(null)` and puts it in context. `MediaAppInner` reads it from context and passes it to `usePlaybackBroadcast`. But `NowPlaying` independently creates its own local `playerRef = useRef(null)` and passes THAT to `MediaAppPlayer` as `ref={playerRef}`. The Player's imperative handle is attached to NowPlaying's local ref. The context ref stays permanently `null`. `buildBroadcastMessage` reads `playerRef.current?.getMediaElement?.()` — always `null` — and returns `null`. No broadcast ever fires. Device monitoring is fully non-functional.

`handlePrev` in `MediaAppInner` also reads `playerRef.current?.seek?.(0)` from the context ref — that seek never executes either.

**Evidence:**
- `MediaAppContext.jsx:9` — `const playerRef = useRef(null)` created in context
- `MediaApp.jsx:29,31` — `playerRef` destructured from context, passed to `usePlaybackBroadcast`
- `NowPlaying.jsx:59` — separate `const playerRef = useRef(null)` created locally
- `NowPlaying.jsx:192` — `ref={playerRef}` uses the local ref, not the context ref
- `usePlaybackBroadcast.js:22` — reads `playerRef.current?.getMediaElement?.()` — always null
