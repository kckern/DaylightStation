# Cycle Game — Audit Phases 1+2 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. One task per implementer; task briefs cite the audit (`docs/_wip/audits/2026-07-01-cycle-game-audit.md`) as spec. Implementers MUST read the target files before editing — this plan pins decisions and acceptance criteria, not verbatim code.

**Goal:** Phase 1 (trustworthy racing: wall-clock timing, interpolated finishes + dead heats, overtime vs DNF, sensor-gap cap, crash recovery) and Phase 2 (readable racing: one motion clock, honest chart, standings tower, POV that never loses anyone, speedometer geometry, 10-foot type, palette consolidation).

**Architecture:** Engine/controller stay pure (all clocks injected). Presentation motion moves to a single shared rAF interpolation pattern (PovGrid's proven model). New UI is prop-driven panels. No backend changes.

## Global Constraints

- Structured logging only (`getLogger` children / `uiLog`); never raw console.*.
- Tests: `./node_modules/.bin/vitest run --config vitest.config.mjs <paths>` from worktree root; NEVER mix backend+frontend paths in one invocation; no skipped/vacuous assertions; capture the runner's own summary.
- Engine (`CycleRaceEngine.js`) and controller (`CycleRaceController.js`) must remain clock-free and DOM-free (inputs injected).
- React MUST NOT re-render per animation frame in any panel; rAF loops write DOM/refs imperatively (PovGrid pattern). React renders per 1 Hz tick.
- All existing tests must stay green per task (run the full `widgets/CycleGame/ + lib/cycleGame/` sweep before each commit); update tests only when the SPECCED behavior changed, never to paper over regressions.
- Race-screen text floor: nothing below 1.1rem may render during `racing` (T11 enforces; earlier tasks must not add new sub-floor text).
- Commit per task: `fix(cycle-game): …` / `feat(cycle-game): …`.
- The persisted race-record schema (`raceRecord.js`) is frozen except where a task explicitly extends it.

## Phase 1 — trustworthy racing

### T1: Wall-clock race time + one interval across go→racing
**Audit:** F8 (major). Tick-counted clock drifts under kiosk jank; interval torn down at go→racing.
**Files:** `CycleGameContainer.jsx` (race-tick effect ~:869-1028 region), test additions in `CycleGameContainer.test.jsx` or a new focused test.
**Decisions:**
- Engine unchanged (still advances `intervalSeconds` per `tick()`); the CONTAINER anchors wall time: at green (engine-live start) record `raceStartMs = performance.now()`; on each interval fire, compute `dueTicks = floor((now - raceStartMs)/intervalMs) - tickedCount` and run the tick body `dueTicks` times (catch-up loop, hard cap 30 per fire to bound a resumed-from-freeze burst; log a `cycle_game.tick_catchup` warn when dueTicks > 1).
- Effect gates on a derived boolean `isEngineLive = phase === 'go' || phase === 'racing'` so the go→racing flip does NOT tear down/recreate the interval. The comment "set up once per racing phase" becomes true.
**Acceptance:** a simulated late-firing interval (fake timers) still yields engine elapsed == wall elapsed ±1 tick; go→racing transition does not reset the interval (assert via effect-instrumentation or tick continuity); all existing container/engine tests green.

### T2: Interpolated finish times + dead-heat rule
**Audit:** game-design #8. `finishTimeS = elapsedS` quantized to 1 s; ties decided by Map insertion order.
**Files:** `lib/cycleGame/CycleRaceEngine.js` (finish stamping ~:122-125, `standings()` ~:157-163), `CycleRaceEngine.test.js`, `RaceResults.jsx` only if placement-rendering needs a tweak for shared placement.
**Decisions:**
- Stamp finish by within-tick interpolation exactly like lap splits three lines below: `finishTimeS = elapsedS - intervalS + frac*intervalS`, `frac = (goalM - prevCumM)/(cumM - prevCumM)` (guard div-by-zero → frac 1).
- `standings()`: finished riders sort by interpolated `finishTimeS`; dead heat = |Δt| < 0.05 s → SAME placement, next placement skips (1,1,3). Unfinished riders keep distance ordering after finishers.
- RaceResults: two riders with placement 1 both get 🥇 (verify the existing `MEDALS[placement]` path renders this; adjust only if broken).
**Acceptance:** engine tests: (a) two riders crossing in the same tick get distinct interpolated times and correct order; (b) within 50 ms → shared placement + skip; (c) lap-split behavior unchanged. Persisted `final_time_s` now carries the interpolated value (raceRecord test updated if it asserts integers).

### T3: Overtime ≠ DNF
**Audit:** game-design #7 (mercy-kill brands honest finishers "DNF", hides their distance).
**Files:** `lib/cycleGame/CycleRaceController.js` (mercy-kill ~:147-160, `finishNow`), `CycleRaceController.test.js`, `CycleGameContainer.jsx` (snapshot plumbing), `RaceResults.jsx/.scss/.test.jsx`.
**Decisions:**
- Controller gains an `overtime` set, exposed in snapshots alongside `dnf`. Mercy-kill and `finishNow` cuts go to `overtime`; idle-quit stays `dnf`. Ghosts exempt as today.
- RaceResults: overtime rows show their REAL metric (distance for distance races — e.g. "2.1 km") plus a small "OT" tag; placement continues after finishers by distance. DNF rows unchanged. Legend gains "OT — still riding when the race closed". Event toast copy for mercy-kill updated from DNF phrasing if it says DNF.
**Acceptance:** controller tests: mercy-killed rider lands in overtime (not dnf), idle rider still dnf, ghost still exempt; RaceResults test renders an overtime row with metric + tag and keeps DNF rendering.

### T4: Sensor-gap hold cap + "sensor lost" chip
**Audit:** game-design #6 (a dead sensor rides forever at frozen RPM and can never DNF).
**Files:** `lib/cycleGame/equipmentRpm.js` + test, `CycleGameContainer.jsx` (gap accounting in the tick input resolution ~:886-910 region), `CycleSpeedometer.jsx/.scss` (+ its test) for the chip.
**Decisions:**
- `rpmDuringGap(recentRpms, gapTicks)`: ticks 1-5 of a gap behave as today (hold w/ cooldown heuristic); ticks 6-8 decay the held value by ×0.5 per tick; tick ≥9 returns 0. Pure, tested.
- Container tracks consecutive gap ticks per rider (reset on any connected reading) and passes it; when gapTicks ≥ 9 the rider's `riderLive` entry gains `sensorLost: true`.
- CycleSpeedometer: small "SENSOR" chip (≥1.1rem, `$cg-warn`-style color) shown when `sensorLost`; hidden otherwise. An `hr_change`-style edge log `cycle_game.sensor_lost` fires once per episode (mirrors `cadence_change`).
- With the hold expiring to 0, the existing idle-DNF clock can now fire for a truly dead sensor — that is the intended outcome; assert it in a controller-level or container-level test.
**Acceptance:** equipmentRpm tests for hold/decay/zero phases + cooldown heuristic preserved; chip renders on flag; edge log fires once.

### T5: Mid-race crash recovery (checkpoint → finalize)
**Audit:** C1 (critical): reload mid-race loses the race entirely.
**Files:** new `lib/cycleGame/raceCheckpoint.js` + test; `CycleGameContainer.jsx`; container test.
**Decisions:**
- Every 5th tick during go/racing, write `sessionStorage["cycleGame.checkpoint"]` = `{ raceMeta, engineState (already-serializable clone from getState()), savedAt: Date.now() }`. Clear on: normal results save success, cancel, and after recovery.
- On container mount (entering idle), if a checkpoint exists and `savedAt` within 30 min: FINALIZE it — build the record (`buildRaceRecord(engineState, raceMeta)`) and persist via `saveRaceRecord` (zero-distance guard applies), log `cycle_game.race_recovered`, show a toast/info on the lobby ("Recovered your interrupted race — saved to history"), clear the checkpoint. Stale/invalid checkpoints are cleared silently with a debug log. (Live resume of the running engine is explicitly out of scope — finalize only.)
- `pagehide`/`beforeunload` during go/racing: best-effort `navigator.sendBeacon('/api/v1/fitness/cycle-races', new Blob([JSON.stringify({record})], {type:'application/json'}))` with the CURRENT engine state; keep the checkpoint regardless (belt over the beacon's braces — double-submit is safe, same raceId overwrites).
- Pure helpers in `raceCheckpoint.js`: `writeCheckpoint(store, data)`, `readFreshCheckpoint(store, nowMs, maxAgeMs=30min)`, `clearCheckpoint(store)` — injected storage for tests.
**Acceptance:** lib tests (fresh/stale/corrupt/missing storage); container behavior: checkpoint written during racing (fake timers), recovery path saves + clears + logs; normal save clears checkpoint.

## Phase 2 — readable racing

### T6: One motion clock (kill the desync + the 60 fps chart re-render)
**Audit:** C4 (the "SVG drags behind" complaint) + F11 (DistanceChart 60 fps setState).
**Files:** new `lib/cycleGame/motionClock.js` + test; `panels/DistanceChart.jsx` (major), `CycleRaceScreen.scss` (remove data-motion transitions ~:100-107, :125-129), `CycleSpeedometer.jsx/.scss` (needle + odometer), `panels/OvalTrack.jsx/.scss` (markers), tests for each.
**Decisions:**
- `motionClock.js`: `createTickLerp({ intervalMs })` returning `{ onTick(payload), subscribe(cb), stop() }` — one rAF loop per panel instance, linear fraction 0→1 across the tick interval (reuse/absorb `tickFraction.js` semantics; keep that module's exports working or migrate its consumers).
- DistanceChart: geometry (line coords, grid, tag layout) computed per TICK in `useMemo`; the leading-edge glide, terminus tags, connectors, and event markers are positioned per FRAME by imperative ref writes from the clock callback. `setTickFrac`-style per-frame state is REMOVED. Decimate plotted points beyond ~600 per rider (plot every Nth) so geometry cost is bounded.
- Delete the CSS `transition: left/top` rules for chart tags/markers/connectors; delete the needle's `transition: transform 0.18s`; delete oval markers' 0.9 s transition. All three now lerp on the shared clock (needle rotation, oval marker x/y, odometer count-up).
- Easing: LINEAR everywhere for data motion (the data is a rate; easing adds false dynamics). CSS animations remain ONLY for state transitions (enter/exit, penalty pulse, results fly-in).
**Acceptance:** grep proves no `transition:` on the removed selectors and no per-frame setState in DistanceChart; all panel tests green (updated where they asserted the old mechanics); a motionClock unit test drives fake rAF and asserts linear interpolation + stop() cleanup. THE RISKIEST TASK — implementer must eyeball the math (positions at fraction 0 and 1 must equal the previous/current tick positions exactly).

### T7: The chart answers "how much is left"
**Audit:** UX §2.1-2.3, 2.6-2.7 (no axis labels; goal line hidden ~85% of race; 2× zoom rug-pulls; header waste).
**Files:** `panels/DistanceChart.jsx/.scss` (+tests), `lib/cycleGame/chartScale.js`/`chartZoom.js` (+tests) as needed.
**Decisions:**
- DISTANCE races: Y domain FIXED [0..goalM] from tick 1; goal line + "🏁 goal" label always visible at top; NO Y zoom steps at all. X domain: keep existing time windowing but rescale CONTINUOUSLY (ease the window edge per tick on the motion clock — no 2× snaps).
- TIME races: keep auto-zoom but continuous (same eased-window approach) — no stepped doubling; the lin↔log flip tweens over ~400 ms and shows a small "zoomed on leaders" chip while log is active.
- Axis labels (HTML overlay, ≥1.1rem): 2-3 Y labels (distance values; in log mode label as gap-behind-leader), 2-3 X labels (m:ss). Header band shrinks to auto height; race clock grows to ≥2.4rem; "`1.23 km` led" copy becomes "Leader 1.23 km".
- Terminus tags: replace the single-initial disc with a 28px avatar image (same identity resolution the speedo row uses) + gap-behind-leader text ("−42 m"); leader shows total distance.
**Acceptance:** tests for fixed-Y mapping (rider at goal renders at top edge from tick 1), axis label values, continuous window (no 2× jumps between consecutive ticks), tag content. Chart still renders empty-series gracefully.

### T9: A POV that never loses anyone
**Audit:** C5/UX §1 (leader fogged out + shrunk at big gaps; trailing rider rides an unlabeled road; empty at GO; no rank/gap badges).
**Files:** `lib/cycleGame/povWorld.js` + test, `lib/cycleGame/povFollowCam.js` + test if touched, `panels/PovGrid.jsx/.scss` + test.
**Decisions:**
- Marks/gates generate across `[lastPlaceM - 30, leaderM + aheadM]` (rider-anchored, not leader-anchored) — povWorld change with tests.
- Gap compression: rider z beyond ~100 m ahead of the camera anchor compresses via `100 + 40*log1p((gap-100)/40)` (pure fn in povWorld, tested) so the leader always renders on-screen; card min scale floor raised so a compressed far leader stays ≥0.45 scale.
- When the TRUE gap exceeds the compressed window, additionally pin a fixed-size horizon chip: "LEADER +312 m" (pooled DOM label, chart-style plate, ≥1.2rem).
- Start-line lineup: all riders (incl. ghosts) render parked at z=0 from mount; the `distM > 0` filter applies only to CAMERA framing after a 5 s grace. Initial `setLookAt` fires on mount so the first frame is framed.
- Every card gets a fixed-screen-size badge (NOT depth-scaled): rank ordinal + gap-to-next ("2nd · −12 m"), ≥1.2rem, lane-colored plate; the depth-scaled absolute-distance label under the avatar is removed (superseded).
**Acceptance:** povWorld tests (span, compression monotonic + identity ≤100 m); PovGrid test updated for lineup-at-zero; manual math check that fraction-0/1 positions match ticks (motion unchanged from T6's pattern — PovGrid already does this right).
